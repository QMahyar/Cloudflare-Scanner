## Why this matters

Small consistency papercuts that compound: the failure DTO is declared identically
as a function-local `type failEntry struct { Endpoint string; Error string }` in
three handlers (`server.go:987`, `server.go:454`-area for WARP results, and
`server.go:1454` in clean results). And the two scanner tabs throttle their
result-fetch at *different* intervals for no stated reason — `EndpointScanner.svelte`
uses `600 - (Date.now()-lastFetch)`, `IpScanner.svelte` uses `500`. Neither is a bug,
but the divergence is the kind of thing that quietly grows. Hoisting one shared
`failEntry` type and picking a single throttle constant removes two "edit-in-N-places"
traps.

Note: the `resultEntry` structs are NOT identical across handlers (the clean-IP one
carries colo/H3/score/loss columns the WARP one doesn't), so do NOT try to merge
those — only the `failEntry` DTO is genuinely shared.

## Steps

### Step 1: Hoist `failEntry`

Declare once at package scope (near the other response DTOs, or in `httpserver.go`):
```go
// failEntry is the shared failure row shape for scan/clean results responses.
type failEntry struct {
    Endpoint string `json:"endpoint"`
    Error    string `json:"error"`
}
```
Remove the two function-local `type failEntry struct {...}` declarations; the code
that builds `[]failEntry{...}` continues to compile against the package type
unchanged.

**Verify**: `go vet ./...` → exit 0; `go build` → exit 0; `go test ./...` → ok.

### Step 2: Pick one throttle interval

Choose a single value (recommend 600ms to match the heavier endpoint table, or 500ms —
either is fine, just make them equal) and set both components' `scheduleFetch` interval
to it. Keep IpScanner's `force` fast-path. Add a one-line comment noting the shared value.

**Verify**: `grep -n "Date.now() - lastFetch" frontend/src/components/*.svelte` shows the same constant in both.

### Step 3: Rebuild the UI bundle

```
cd frontend && npm run build && cd ..
go build -ldflags="-s -w" -o /dev/null .
```

**Verify**: both exit 0.
